# 2026-04-09

## Summary

- Scaffolded the first Cargo workspace for `ssw-rs`.
- Added `ssw-core`, `ssw-actix`, `ssw-html`, and `ssw-components`.
- Implemented a minimal compileable API surface and verified it with `cargo test`.

## Facts

- `ssw-core` currently defines `Render`, `HtmlKind`, `HtmlResponse`, `TextResponse`, `Redirect`, and `Response`.
- `ssw-html` currently uses a small owned `Markup` type with escaping helpers and simple document composition.
- `ssw-actix` currently exposes response conversion helpers rather than a more ambitious adapter abstraction.
- `ssw-components` currently contains only thin proof-of-shape helpers.

## Risks

- The current HTML layer is intentionally too small to judge long-term ergonomics.
- `Render` may prove too weak or too general once layouts, assets, and partial rendering become more concrete.
- No example app exists yet, so the current API has only unit-test pressure.

## Additional work

- Added an initial `html!` macro with support for nested elements, string literals, escaped expressions, boolean and valued attributes, `@if`, and `@for`.
- Kept the macro implementation behind an internal `ssw-html-macros` crate while re-exporting the public API from `ssw-html`.
- Added tests that define the intended authoring style instead of optimizing only for low-level rendering helpers.

- Expanded `ssw-html` with empty-tag syntax, `#id` and `.class` shorthand, optional attribute omission via `Option<T>`, and a `Document` builder for page layouts.
- Added layout-shaped tests so `ssw-html` is being driven by real page authoring patterns rather than just raw string generation.
- Improved class composition so `.class` shorthand now merges with explicit `class=(...)` values, including optional and tuple-composed values.
- Current limitation: `#id` shorthand still does not compose with explicit `id=...` attributes.

- Added an end-to-end Actix test flow that serves a document page, an HTML fragment, and redirects using the current `ssw-html` API.
- Added thin `ssw-actix` convenience helpers (`page`, `fragment`, `redirect`) and `Responder` support for `ActixResponse` to make handler code read like the framework model.

- Fixed a real `ssw-html` semantic bug: expression-based boolean HTML attributes now render by presence or omission for standard boolean attrs, while non-boolean attrs still render string values.

- Added an end-to-end POST form flow in Actix with invalid redisplay, preserved input values, and redirect-after-success behavior.
- Confirmed that the current `ssw-html` surface is already usable for simple SSR form workflows without requiring a separate form abstraction yet.

- Expanded the form flow to field-level validation with per-field errors and accessible markup (`aria-invalid`, `aria-describedby`), while still preserving submitted values.
- Confirmed that the current stack can express realistic SSR validation UX without introducing a generic form layer yet.

- Added doc comments across the public crate surfaces and updated the README to describe the implemented slice rather than only the original roadmap.
- Rewrote the small `ssw-components` alert to use `html!`, keeping even optional components aligned with the same HTML authoring model.
